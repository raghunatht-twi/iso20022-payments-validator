Overview

The ISO 20022 Payments Test Automation Platform is a three-layer system that automates the full lifecycle of ISO 20022 message testing — from synthetic test data generation through event-driven pipeline execution to auditable reconciliation reporting. It covers the pain (Payments Initiation) domain across all 12 active message sets (001–018).

The platform combines an AI agent (powered by Claude), a multi-threaded Kafka pipeline, an XSD validation engine (lxml), Ed25519 digital signature integrity checking, and a SQLite state store to eliminate the manual effort of creating, processing, and proving completeness of ISO 20022 test suites.

Architecture Layers

1
Test Data Generation
generate_test_data.py — AI-powered synthetic XML generation

An AI agent reads each XSD schema, extracts the target namespace, root element, and format constraints, then prompts Claude to generate a corpus of ~50 XML test messages per schema. Every generated message is immediately validated against the source XSD by lxml, and re-categorised based on the actual validation result — not the model's stated intention. 

  XSD Schema (12 files)

      │                              format rules (IBAN, BIC, date, UETR)

      │                              category instructions (pass/fail/edge)
      │
      ├─ _call_claude()              Stream response from Claude API
      │                              claude-sonnet-4-6 (default)
      │
      ├─ _extract_json()             Parse JSON array from model response
      │
      ├─ etree.fromstring()          Parse each XML with safe parser
      ├─ schema.validate()           Validate against XSD
      │
      └─ _actual_category()          Re-bucket: pass / fail / edge
              │
              ▼
  test_data/pain/{001…018}/
      gen-pass-NNN.xml   (35 per schema — valid messages)
      gen-fail-NNN.xml   (10 per schema — deliberate violations)
      gen-edge-NNN.xml   ( 5 per schema — boundary conditions)

The 70 / 20 / 10 distribution ensures systematic coverage: the pass set exercises realistic payment data diversity; the fail set verifies that validators correctly reject non-conformant messages; the edge set probes boundary conditions such as maximum field lengths and Unicode characters in name fields.

2
Kafka Pipeline + Message Integrity
sender_agent.py → Kafka → receiver_agent.py — signed messages, tamper detection, exactly-once semantics

The pipeline streams every XML test message through Kafka, validates it against its schema, and routes failures to a dead-letter queue. Every message is Ed25519-signed by the sender; the receiver verifies the signature before any XSD validation. Messages that fail signature verification are quarantined to a dedicated tampered topic. A multi-threaded receiver — one thread per domain/message-set — processes all schemas in parallel. The combination of Kafka's at-least-once delivery and SQLite's idempotent writes produces exactly-once processing semantics without requiring Kafka transactions.

Reporting

Two independent reporting scripts produce self-contained, Thoughtworks-branded HTML artefacts. The validator report can be run at any time against any schema. The reconciliation report reads the live SQLite state and produces a timestamped audit document proving processing completeness.

Components

generate_test_data.py
AI Agent

Reads XSD schemas, calls Claude API, generates 50 XML test messages per schema across three categories. Validates all AI output with lxml before saving. Produces a generation summary HTML report.

ISO20022_validator.py
Validation Engine + Reporter

Validates any set of XML files against a specified XSD schema. Produces a Thoughtworks-branded HTML report with per-file PASS/FAIL, precise error locations, and suggested fixes. Hash-checks known schemas before loading.

generate_keys.py
Key Generation

Run once before the first pipeline execution. Generates an Ed25519 key pair: keys/sender_private.pem (chmod 0o600 — owner only) and keys/sender_public.pem. Refuses to overwrite an existing private key to prevent accidental rotation. Both files are gitignored.

sender_agent.py
Kafka Producer + Signer

Discovers all XML test files, computes content-addressed SHA-256 message IDs, signs each message with the Ed25519 private key, and publishes to per-schema Kafka topics. Idempotent — skips files already recorded in sent_messages. Creates topics (including iso20022.tampered) if they do not exist.

receiver_agent.py
Kafka Consumer + Verifier + Validator

Spawns one daemon thread per schema. Each thread polls its topic, checks for duplicates, verifies the Ed25519 signature (routes tampered messages to iso20022.tampered), validates XML with lxml, records results in SQLite, and forwards XSD failures to the DLQ. Manual offset commit guarantees exactly-once.

reconciliation_report.py
Audit Reporter + Analytics

Reads state.db via SQLite for per-message status, then runs DuckDB analytical queries (schema breakdown, error patterns, test category vs outcome, tampered by schema) against the same file via DuckDB's native SQLite attachment. Produces a single timestamped HTML report combining reconciliation proof, tampered message log, and analytics.

docker-compose.yml
Infrastructure

Defines a single-broker Confluent Kafka 7.6 instance in KRaft mode (no Zookeeper). Exposes two listeners: localhost:9092 for host-machine scripts and kafka:29092 for Docker-internal services. Includes a healthcheck before dependent services start.

Kafka UI (provectuslabs)
Monitoring Dashboard

Visual web dashboard at http://localhost:8080. Shows all topics with message counts, lets you browse individual messages (including DLQ payloads), monitors consumer group lag per partition, and displays real-time throughput graphs. Starts automatically after Kafka's healthcheck passes.

Data Flows

End-to-end flow

1
Schema discovery — generate_test_data.py scans schema/pain/*/ and finds all XSD files.
2
Prompt construction — for each schema, the namespace URI, root element tag, and format constraints are extracted and embedded in a structured prompt alongside the raw XSD content.
3
AI generation — Claude streams a JSON array of XML strings. The response is parsed and each XML is extracted.
4
Validation + categorisation — each XML string is parsed by lxml and validated against the XSD. The actual validation result determines the file category (gen-pass, gen-fail, gen-edge), overriding the model's stated category.
5
File persistence — validated XML files are written to test_data/pain/{msg_set}/ with sequential filenames.
6
Kafka publish — sender_agent.py reads each XML file, computes its SHA-256 message ID, signs the XML bytes with the Ed25519 private key, and publishes the payload (including the base64 signature) to the appropriate Kafka topic. The send is recorded in sent_messages.
7
Signature verification — each receiver thread extracts the signature field and calls Ed25519PublicKey.verify(). A missing or invalid signature routes the message to iso20022.tampered and records it in tampered_messages. XSD validation is skipped for tampered messages.
8
XSD validation — if the signature is valid, the receiver validates the XML content against the XSD. Results are written to processed_messages; failures are forwarded to iso20022.dlq.
9
Reconciliation — reconciliation_report.py reads all four SQLite tables and produces an HTML audit report with DuckDB analytics. A "Fully reconciled" verdict is issued when every row in sent_messages has a corresponding row in processed_messages.

Exactly-Once Processing Guarantee

Exactly-once semantics are achieved by combining Kafka's at-least-once delivery guarantees with application-level idempotency in SQLite — without requiring Kafka transactions or a distributed coordinator. 

  GUARANTEE LAYERS

  Layer 1 — Content-addressed message ID
    message_id = sha256( domain + "." + msg_set + ":" + xml_bytes )
    Same file → same ID, always.
    Different file same name in different schema → different ID (scope prefix prevents collision).

  Layer 2 — Idempotent database write
    INSERT OR IGNORE INTO processed_messages VALUES (message_id, ...)
    Second INSERT for the same message_id is silently discarded by SQLite.

  Layer 3 — Offset commit after DB write
    consumer.poll()  →  validate  →  INSERT OR IGNORE  →  consumer.commit()
                                                              ↑
    Crash here ───────────────────────────────── Kafka redelivers message
    (redeliver is caught by Layer 2 as a duplicate — recorded in duplicate_events)

  Layer 4 — Duplicate audit trail
    Every re-delivery is recorded in duplicate_events (not silently dropped).
    Reconciliation report shows "Duplicates Caught" separately from processed messages.

  RESULT: every message in sent_messages appears exactly once in processed_messages
  regardless of restarts, redeliveries, or re-runs of sender_agent.
ScenarioWhat happensOutcome

Message Integrity — Ed25519 Digital Signatures

Every message traversing the Kafka pipeline is cryptographically signed by the sender and verified by the receiver. This ensures that any tampering of message content in transit is detected before XSD validation runs. 

  INTEGRITY FLOW

  Key Generation  (uv run generate_keys.py — once)
    Ed25519PrivateKey.generate()
      ├─ keys/sender_private.pem   ← sender only, chmod 0o600, gitignored
      └─ keys/sender_public.pem    ← distributed to receiver

  Sender  (for every XML file)
    xml_bytes = xml_path.read_bytes()
    signature = base64.b64encode( private_key.sign(xml_bytes) ).decode()
    payload["signature"] = signature
    producer.send(topic, value=payload)

  Receiver  (first check on every message)
    sig_bytes = base64.b64decode( payload["signature"] )
    public_key.verify( sig_bytes, xml_content.encode("utf-8") )
      │
      ├─ raises InvalidSignature → TAMPERED
      │     forward to iso20022.tampered
      │     INSERT tampered_messages
      │     commit offset
      │     (skip XSD validation entirely)
      │
      └─ success → VERIFIED
            proceed to XSD validation
PropertyValue
AlgorithmEd25519 (Edwards-curve Digital Signature Algorithm)
LibraryPyCA cryptography >=42.0,<46.0
Signed payloadRaw XML bytes (xml_path.read_bytes()) — not the JSON envelope
Signature encodingBase64 (URL-safe string in Kafka JSON payload)
Key storagekeys/ directory — gitignored; private key chmod 0o600
Key formatPEM-encoded PKCS#8 (private) / SubjectPublicKeyInfo (public)
Tamper destinationiso20022.tampered Kafka topic + tampered_messages SQLite table
Tamper reasonForwarded in payload as "tamper_reason": "signature mismatch" or "missing signature"
Why Ed25519? Ed25519 uses 32-byte keys and 64-byte signatures, is immune to timing side-channels, provides 128-bit security, and is considerably faster than RSA-2048 or ECDSA-P256 for both signing and verification — making it the right choice for high-throughput message pipelines.

Key Design Decisions

DecisionAlternatives consideredRationale
One Kafka topic per schema Single topic with schema discriminator field Independent receiver threads — a failing schema does not block others. Natural consumer group partitioning. Topic name encodes the schema identity explicitly.
SHA-256 as message ID (content-addressed) UUID generated at send time; file path as ID Same file content always produces the same ID regardless of when or how many times it is sent. Scoped with domain+msg_set prefix to prevent cross-schema ID collisions.
SQLite for state (not Kafka itself) Kafka consumer group offsets only; Redis; Postgres Zero infrastructure beyond Kafka. SQLite WAL mode handles concurrent multi-thread writes. Queryable with standard SQL for reconciliation and inspection.
Offset commit after DB write Auto-commit; commit before write Guarantees at-least-once delivery. Combined with INSERT OR IGNORE, produces exactly-once processing without Kafka transactions.
lxml re-validates all AI output Trust the model's stated category LLM output is treated as untrusted. Ground truth is the XSD validator. Mismatch counts surface model accuracy without blocking the pipeline.
Schema facts extracted and injected into prompt Send raw XSD only; send no schema context Reduces hallucination on schema-specific rules (namespace URI, element ordering). Injecting concrete facts (IBAN pattern, BIC format) significantly reduces recategorisation rate.
KRaft mode for Kafka (no Zookeeper) Kafka + Zookeeper (classic mode) Simpler docker-compose (one container). KRaft is the default from Kafka 3.3+. Eliminates the Zookeeper failure domain for local development.
PEP 723 inline script dependencies requirements.txt; pyproject.toml; Pipfile Each script is self-contained — no shared virtual environment setup. uv run script.py installs exactly what that script needs and nothing more.
Ed25519 for message signing (not HMAC, not RSA) HMAC-SHA256 shared secret; RSA-2048; ECDSA-P256 Ed25519 uses asymmetric keys so the receiver needs no secret — the public key is safe to distribute. 32-byte keys, 64-byte signatures, immune to timing side-channels, and significantly faster than RSA or ECDSA at equivalent security levels.
Sign raw XML bytes, not the JSON envelope Sign the full serialised JSON payload XML bytes are stable — the same file always produces the same bytes. The JSON envelope could be legitimately re-serialised with different field ordering, which would break signature verification. Signing the canonical input (XML) avoids this class of false positives.

Technology Stack

Python 3.11+
Application Runtime

All five scripts. Uses from __future__ import annotations, dataclasses, type annotations, threading, signal handling, and pathlib throughout.

uv
Package Manager

Replaces pip + venv. PEP 723 inline metadata means each script declares and installs its own dependencies on first run. No global environment to manage.

lxml 5.x
XML / XSD Engine

Wraps libxml2 + libxslt. Provides etree.XMLSchema for XSD validation, etree.XMLParser with hardened flags, and schema.error_log for precise error messages.

Anthropic Python SDK
LLM Client

Streaming Messages API. Adaptive thinking enabled automatically for Opus/Fable models. Default model: claude-sonnet-4-6. Override via --model= CLI flag.

Apache Kafka 7.6 (Confluent)
Message Broker

KRaft mode — no Zookeeper. Single broker, 3 partitions per topic, replication factor 1. Consumer group: iso20022-receivers. DLQ topic: iso20022.dlq.

SQLite (WAL mode)
State Store

Four tables: sent_messages, processed_messages, duplicate_events, tampered_messages. WAL mode enables concurrent writes from 12 receiver threads. Parameterised queries throughout.

PyCA cryptography 42+
Message Integrity

Ed25519 key generation (Ed25519PrivateKey.generate()), signing (private_key.sign(data)), and verification (public_key.verify(sig, data) raises InvalidSignature on tamper). Keys serialised as PEM-encoded PKCS#8 / SubjectPublicKeyInfo. Installed automatically via PEP 723 in generate_keys.py, sender_agent.py, and receiver_agent.py.

DuckDB 1.x
Analytics Engine

Attaches directly to state.db via ATTACH '...' AS state (TYPE sqlite) — no data copying or import step. Runs three analytical queries in reconciliation_report.py: per-schema pass rate breakdown, validation error pattern ranking, and test category vs actual outcome cross-tabulation. Third-party library installed automatically via PEP 723 inline metadata.

Kafka UI (provectuslabs)
Monitoring Dashboard

Runs as a Docker service alongside Kafka. Available at http://localhost:8080. Connects to Kafka via the PLAINTEXT_INTERNAL listener (kafka:29092) — the Docker-internal address — to avoid the hostname resolution issue that occurs when containers use the externally advertised localhost:9092 address.
