The Business Challenge

ISO 20022 is the global standard for financial messaging, mandated by SWIFT, central banks, and payment market infrastructures worldwide. Its adoption is accelerating: SWIFT's cross-border payments network completed migration in 2025, and domestic payment schemes across Europe, the UK, and Asia-Pacific are following suit.

For any financial institution that originates, processes, or validates payments, conformance to ISO 20022 schemas is non-negotiable. The pain (Payments Initiation) domain alone covers 12 distinct message types — from customer credit transfers and direct debits to mandate management — each defined by a strict XSD schema with precise element ordering, enumeration constraints, pattern rules, and cardinality requirements.

AI-Powered Test Data Generation

An agent powered by Claude (Anthropic) reads each XSD schema, extracts the exact namespace URI, element ordering rules, and value constraints, then generates a diverse corpus of test messages across three categories. Every generated message is immediately validated by lxml against the schema and re-bucketed if it does not match its intended category.

✅
Schema-Accurate Validation

The ISO20022 Validator checks any XML message against its corresponding XSD schema using lxml, producing a Thoughtworks-branded HTML report with per-file pass/fail status, precise error locations, and suggested fixes. Supports all 12 pain message sets with a single command.

📨
Event-Driven Kafka Pipeline

The sender agent publishes every test message to a dedicated Kafka topic per domain and message set. Twelve receiver agents — one per message set — consume in parallel, validate each message against its XSD, and route failures to a dead-letter queue. All processing is idempotent and survives restarts without reprocessing.

📊
Reconciliation and Audit

A reconciliation report queries the SQLite state database and produces an auditable HTML report proving that every message sent was processed exactly once. Duplicate messages are detected, recorded, and displayed separately — providing a complete evidence trail for compliance and QA sign-off.

Test Coverage by Design

The generator targets a deliberate distribution across three categories, ensuring that any test suite exercises not just the happy path but also the failure modes that matter most in a payments context.

Category Target Share Count per Schema What It Tests
Pass 70% 35 Valid messages with diverse data combinations — amounts, currencies, parties, dates. Some with all optional fields; some with required fields only.
Fail 20% 10 Deliberately invalid messages, each with a different violation: missing required element, wrong element order, malformed IBAN, invalid enumeration code, amount exceeding decimal limit, missing currency attribute, and others.
Edge 10% 5 Valid messages at boundary conditions: strings at maximum length, amounts with maximum decimal places, all optional fields populated, Unicode characters in name/address fields, maximum repetition of repeating elements.

Each generated XML is validated against the actual XSD by lxml before being saved. Files that do not match their intended category are automatically re-labelled — so the saved corpus is always truthfully categorised, regardless of what the AI model produced.

Exactly-Once Processing Guarantee

In payments systems, processing a message more than once — or missing one entirely — can have serious financial consequences. The pipeline is designed so that every message is processed exactly once, and this guarantee is provable by inspection.

  • Content-addressed message IDs — each message is identified by a SHA-256 hash of its content scoped to its domain and message set. The ID is stable: the same file always produces the same ID, regardless of when or how many times it is sent.
  • Idempotent persistence — the receiver uses INSERT OR IGNORE on a unique primary key in SQLite. Even if Kafka redelivers a message after a crash, the second attempt is silently discarded at the database level.
  • Offset commit after write — Kafka offsets are committed only after the database write succeeds. A crash between validation and commit causes redelivery, which is then caught as a duplicate — no message is ever lost.
  • Duplicate detection log — every additional delivery of a known message is recorded in a separate audit table, so the reconciliation report can distinguish between "processed once" and "received multiple times, processed once".
  • Failed messages preserved — messages that fail XSD validation are forwarded to a dead-letter queue topic and recorded with their full error detail, enabling investigation without blocking the main processing pipeline.

Business Value

Effort Reduction

Creating a single ISO 20022 test message manually requires a qualified analyst to read the XSD, understand element ordering and cardinality rules, construct valid XML, and verify it against the schema. A comprehensive suite covering valid, invalid, and edge cases across 12 message sets could realistically take two to three weeks of specialist effort. This platform produces the equivalent corpus in under five minutes.

Compliance Assurance

Regulators and payment scheme operators increasingly require evidence that implementations have been tested against the full schema, not just common cases. The reconciliation report provides a timestamped, auditable HTML artefact that can be presented as evidence of test completeness — showing every message sent, its validation outcome, and proof that no message was skipped or double-counted.

Accelerated Schema Adoption

ISO 20022 schemas are versioned and updated periodically by the standards body. When a new schema version is published — for example, upgrading from pain.001.001.12 to pain.001.001.13 — the entire test corpus for that message set can be regenerated with a single command. Teams are never waiting on manual test case authoring to begin implementation work.

Early Defect Detection

The deliberate failure cases in the test corpus verify that validators and processing systems correctly reject non-conformant messages. Without systematic negative testing, a system may silently accept malformed messages that should be rejected — a risk that only surfaces during live interoperability testing or, worse, in production.

Reduced Dependency on Specialist Knowledge

ISO 20022 is a complex standard. Deep familiarity with its namespace conventions, xs:sequence ordering rules, external code lists, and pattern constraints is not universally available on delivery teams. By encoding this knowledge in the generator and validator, the platform enables any developer to produce and validate conformant messages without needing to be a standards expert.
